# How to create a new release

Find out about the process that project owners must follow to set expectations and ensure consistency when you are creating a new release.

## Version number

Each release version number must increase incrementally and comply with [Semantic Versioning (SEMVER)](https://semver.org/).

Use the following format `vx.x.x`.

## Determining version number

Versioning should be performed according to the semver spec. Accordingly, the patch version must be used only for non-breaking bug fixes, and new features should be a minor version increase.

Since changes to the Helm chart will change behaviour, any change to the Helm chart must be released in a minor version bump. Patch versions of the container image should be consumable without reviewing the Helm chart for changes.

## Steps an agent or contributor can execute locally

These steps require only a local clone of this repository and standard tooling (`go`, `helm`, `make`, `detect-secrets`).

### 1. Determine the new version

- Dependency or vulnerability fixes only → increment the **patch** version (`v0.x.y` → `v0.x.y+1`)
- New behaviour or Helm chart changes → increment the **minor** version (`v0.x.y` → `v0.x+1.0`)
- Discuss with the team if unsure

### 2. Create a branch

```bash
git checkout main && git pull
git checkout -b prep-vx.x.x
```

### 3. Update the version in all four locations

| File | Field |
|---|---|
| `Makefile` | `VERSION=vx.x.x` |
| `helm/portieris/Chart.yaml` | `version: vx.x.x` |
| `helm/portieris/values.yaml` | `tag: vx.x.x` |
| `CHANGELOG.md` | new section + front-matter (see below) |

### 4. Update CHANGELOG.md

Add a new section immediately after `## v-next`:

```markdown
## vx.x.x

Release: YYYY-MM-DD
- <description of changes>
```

Also update the front-matter at the top of `CHANGELOG.md`:
- `lastupdated:` → today's date (`"YYYY-MM-DD"`)

The `copyright: years:` field covers the file's full history; update the end year if it has changed.

### 5. Update go.mod / go.sum if needed

If dependency versions have changed (e.g. `go get -u ./...` was run):

```bash
go mod tidy
```

### 6. Run the test suite

```bash
make alltests
```

This runs `fmt`, `lint`, `vet`, copyright check, and all unit tests. All must pass before proceeding.

### 7. Run detect-secrets — BEFORE committing or pushing

This step is mandatory.

```bash
detect-secrets scan --update .secrets.baseline
detect-secrets audit .secrets.baseline
```

- If `scan` produces a diff beyond a `generated_at` timestamp change, inspect every new finding carefully in `audit`.
- Only accept a finding as a false positive after confirming it contains no real credential.
- If a real secret is found, remove it from the source and do **not** proceed until the scan is clean.

### 8. Commit and push the branch

```bash
git add Makefile helm/portieris/Chart.yaml helm/portieris/values.yaml CHANGELOG.md go.mod go.sum .secrets.baseline
git commit -m "Prep release vx.x.x"
git push origin prep-vx.x.x
```

### 9. Package the Helm chart

```bash
(cd helm/portieris && ./gencerts)
make helm.package
```

This produces `portieris-vx.x.x.tgz` in the repo root. Keep this file — it is attached as an artifact to the GitHub release later.

> **Note:** The certificate generated by `gencerts` is bundled into the chart. It is documented as suitable for development use only.

## Steps requiring IBM-internal access

The following steps cannot be completed from a local clone of the public repository. They require access to IBM internal systems and are covered in a separate internal runbook.

1. Build and push the candidate container image
1. Run e2e tests against the candidate image
1. Create a PR from `prep-vx.x.x` → `main` and merge it
1. Tag the release: `git checkout main && git pull && git tag vx.x.x -m "release vx.x.x" && git push origin vx.x.x`
1. Build and push the published container image from the release tag
1. Create the GitHub release, attaching `portieris-vx.x.x.tgz` as a release artifact
1. Deploy to internal environments
